## 我想知道apple, linkedin 的 data plane ，spark 在執行那些維護作業時，怎麼把狀態或資訊 persist 下來？

根據來源提供的資訊，Apple 與 LinkedIn 在執行 Apache Iceberg 表格維護作業時，透過不同的機制來持久化（persist）維護任務的狀態與資訊。

### Apple 的維護作業狀態持久化機制
Apple 的表格管理服務（Table Management Service, TMS）將 Spark 視為執行層（Data Plane），並透過以下方式管理狀態：

*   **持久化存儲與 API 暴露**：TMS 系統會將每個維護工作負載（workload）的**生命週期以及系統的每一次變更**持久化到一個「耐用存儲」（durable store）中。這些資訊會透過 API 暴露，讓用戶或其他工具可以查詢維護狀態。
*   **雙向 REST 通訊更新狀態**：在執行過程中，Spark 任務內部會啟動一個輕量級的 **REST 伺服器**，這使得 TMS 控制層能與 Spark 驅動程式（Driver）進行通訊，交換元數據與可觀測性資料。具體而言，Spark 任務會不斷將目前的執行進度與狀態更新回 TMS 服務中。
*   **工作負載管理器 (Workload Manager)**：控制層中的管理器會保存維護運行的歷史記錄、當前狀態以及優先級排序資訊，提供維護作業的完整視圖。

### LinkedIn (OpenHouse/AutoComp) 的維護作業狀態持久化機制
LinkedIn 透過其 **OpenHouse** 控制平台與 **AutoComp** 框架來處理維護狀態：

*   **OODA 模型的階段性產出**：AutoComp 遵循「觀察、導向、決定、行動」（OODA）模型。在這些階段中，系統會生成並持久化特定的資訊清單：
    *   **觀察 (Observe)**：產出 `(id, statistics)` 清單，記錄從系統提取的統計數據。
    *   **導向 (Orient)**：產出 `(id, traits)` 清單，將統計數據轉換為描述表格健康狀況或維護潛力的「特徵」（traits），例如估算的檔案數量減少量或計算成本。
    *   **決定 (Decide)**：產出排序後的任務 `ids` 清單。
*   **事件流與 Kafka**：LinkedIn 系統會執行不掃描數據的 Spark 任務來查詢 Iceberg 元數據表（如快照、清單項等），並將這些**事件流發布到 Kafka**。這些 Kafka Topic 包含提交元數據、分區詳細資訊以及列級別的統計數據，供後續的監控與檢測系統使用。
*   **OpenHouse 目錄**：作為一個「宣告式目錄」，OpenHouse 會儲存用戶定義的「期望狀態」與系統觀察到的「實際狀態」，並透過數據服務進行調和。

### 額外參考：Slack 的持久化做法
雖然您的問題主要針對 Apple 與 LinkedIn，但來源中提到的 Slack 做法可作為技術對比。Slack 的 **"Ice Chipper"** 服務在執行維護時，會將所有活動（如維護開始、結束、成功/失敗、堆疊追蹤等元數據）直接寫入一個**「追蹤表」（tracking table）**，而這個追蹤表本身就是一個 Iceberg 表格。其鎖定機制也是透過對另一個 Iceberg 表格（鎖定表）執行 `MERGE` 操作來完成的。

## 多描述這一塊：事件流與 Kafka：LinkedIn 系統會執行不掃描數據的 Spark 任務來查詢 Iceberg 元數據表（如快照、清單項等），並將這些事件流發布到 Kafka。這些 Kafka Topic 包含提交元數據、分區詳細資訊以及列級別的統計數據，供後續的監控與檢測系統使用。
根據來源資料，LinkedIn 實作的這套「零掃描」（Zero-Scan）元數據監控機制，是其 **OpenHouse** 控制平面中極為關鍵的觀測技術。這套機制的設計核心在於利用 Apache Iceberg 已經計算好並儲存在元數據層的資訊，來達成大規模的資料品質（DQ）與異常檢測。

以下是針對該機制的詳細描述：

### 1. 執行的「零掃描」Spark 任務
LinkedIn 並不執行傳統的 SQL 查詢（如 `SELECT count(*)`）來驗證資料，因為這需要掃描實體資料檔案，對於數十萬張表格來說成本太高。
*   **任務性質**：系統會利用一組預先存在的每表 Spark 維護任務（原本用於系統維護），將其功能擴展為查詢 Iceberg 的**元數據表**（如 `snapshots`、`manifest_entries` 以及數據檔案記錄）。
*   **不掃描資料**：這些任務僅讀取 Avro 格式的 Manifest 檔案與 Snapshot 信息，完全不讀取實際的 Parquet 或 ORC 資料列，因此計算開銷極低。

### 2. Kafka 中的三個核心事件流（Topics）
在 LinkedIn 的架構圖中（見 Source 10 的 Figure 1），收集到的資訊會被發送到三個專門的 Kafka Topic，供不同的下游消費者使用：
*   **提交元數據 (Commit Metadata)**：包含 Snapshot ID、提交時間戳記以及 **「寫入者身份」（Writer Identity）**。這是為了區分哪些變更是來自用戶寫入，哪些是來自系統維護（如 Compaction 或 Sorting），以過濾監控雜訊。
*   **分區層級提交詳情 (Per-partition Commit Detail)**：記錄每個分區中新增或變動的檔案數量與大小。這用於「到達指標」（Arrival Metrics）管線，監控資料是否如期抵達或分區是否過度碎片化。
*   **欄位級別統計數據 (Column-level Stats)**：這是最重要的部分，包含記錄數（record count）、空值計數（null value counts）、以及欄位值的上下界（lower/upper bounds）。

### 3. 下遊的監控與檢測系統
Kafka 之後的消費者會將這些流轉換為時間序列信號，進行以下分析：
*   **異常與偏移檢測 (Anomaly & Drift Detection)**：系統會針對「表-分區-欄位」的組合，自動標記出記錄數異常、空值率偏移或範圍違規。
*   **即時性監控**：相較於傳統需耗時 2-24 小時的掃描式剖析（Profiling），這種元數據驅動的方式可以在幾分鐘內（p50 < 5 分鐘）生成警報。例如，當某個 Dim 表格發生錯誤寫入導致所有記錄變為 inactive 或數據量驟降時，系統能立即偵測到。

### 4. 資源效益與規模
*   **規模**：這套機制已部署在 LinkedIn 超過 **200,000 張表格**（總量達 800+ PB）上。
*   **資源節省**：透過取代掃描式剖析，LinkedIn 減少了約 **50% 的計算資源消耗**（TBhr/day）以及 **45% 的 HDFS 讀取量**。
*   **覆蓋率**：單憑 Manifest 檔案中的統計資訊，就能滿足約 **60% 的使用者定義資料品質規則**（如空值檢查、範圍檢查），若進一步擴充（如使用 Theta Sketch），覆蓋率可提升至 90%。

總結來說，這套 Kafka 事件流機制讓 LinkedIn 從「反應式」的資料檢查轉向了**「持續性」的元數據觀測**，實現了在極低邊際成本下對全量表格的監控。

## Apple, Netflix, Linkedin 對於執行層的spark 有哪些優化？ 像是apple 的t-shirt sizing 就是個具體例子

根據來源資料，Apple、Netflix 與 LinkedIn 在執行層（Data Plane）的 Spark 任務上，針對 Apache Iceberg 的維護與監控實施了多項架構與效能優化，旨在平衡資源消耗、執行效率與系統穩定性。

以下是這三家公司具體的優化措施：

### 1. Apple：資源分級與動態擴展
Apple 的表格管理服務（TMS）重點在於解決大規模環境下的資源浪費與隔離問題：
*   **T-shirt Sizing（T-shirt 尺寸化策略）**：這是 Apple 最具代表性的優化，透過調整 **Spark 配置（如 Executor 的記憶體大小與數量）** 將任務分為不同規格（例如 S、M、L），以匹配不同維護任務（Workload）的成本，避免小任務過度撥補資源。
*   **共享 Spark 任務（Sharing Spark Jobs）**：為了減少啟動數以千計獨立 Spark 任務產生的**引導延遲（Bootstrapping latency）**，Apple 讓同一個 Catalog 內的表格共享 Spark 任務，從而大幅降低資源足跡。
*   **動態資源調整**：在 Spark 任務內部使用 **動態資源分配 (DRA)**，並根據中央隊列的任務積壓情況（Queue length）動態啟動更多 Spark 實例。
*   **雙向 REST 通訊**：在 Spark 任務內啟動輕量級 **REST 伺服器**，讓控制層能即時獲取 Spark 驅動程式的執行進度與觀察指標。

### 2. Netflix：異步清理與元數據優先遷移
Netflix 透過建立高度專業化的 Spark 進程池來處理數據生命週期：
*   **Janitors（清理進程池）**：將 Spark 運作為一組**定期調度的進程池**，專門執行 TTL 清理、快照過期與孤兒檔案刪除任務，並透過 SQS 進行解耦以防過度刪除。
*   **元數據式遷移（Metadata-only Migration）**：開發了 Spark SQL Procedure，在從 Hive 遷移至 Iceberg 時，僅在現有檔案上**疊加元數據**而非複製數據，極小化資料移動成本。
*   **安全存取優化（Signer Service）**：在 Spark 驅動程式中整合外部 **Signer Service**，動態獲取 STS Token，讓 Spark 能安全地存取受保護的 S3 路徑，而無需處理低階 IAM 權限。

### 3. LinkedIn：混合範圍與零掃描觀測
LinkedIn 的 **OpenHouse** 與 **AutoComp** 框架則在任務粒度與監控成本上有深度優化：
*   **混合範圍策略（Hybrid Scoping Strategy）**：對於大型表格，系統會自動選擇**分區級別（Partition-scope）**而非表格級別的壓縮，這能增加並行度、減少資源爭用，並提高容錯能力（失敗時無需重跑整張表）。
*   **零掃描數據品質監控（Zero-Scan DQ）**：這是一項顯著優化，Spark 任務**僅查詢 Iceberg 的元數據表**（如 snapshots, manifests），而完全不掃描實體數據列，這為 LinkedIn 節省了約 **50% 的計算資源**。
*   **多目標優化排序（MOOP）**：在資源受限的情況下，Spark 任務的執行順序是透過 MOOP 算法確定的，該算法會對**「檔案減少量」與「計算成本（GBHr）」**進行權衡，優先執行 ROI 最高的任務。
*   **基於變更的同步（Shadowing）**：在遷移過程中，利用 Iceberg 的元數據層僅針對**變更過的分區**進行增量同步，維持 Hive 與 Iceberg 的一致性。

這些優化共同展示了從「自動化」向「智慧化」演進的趨勢，特別是在處理數十萬張表格的規模時，透過**任務分級、解耦執行與元數據優先處理**來極大化營運效率。

## 哪些是event driven的方式在觸發這些table maintenance job?

根據來源資料，Netflix 與 Apple 在管理 Apache Iceberg 表格時，除了傳統的定時排程外，都引入了**事件驅動 (Event-driven)** 的方式來觸發維護任務。以下是具體的觸發方式與機制：

### 1. Netflix 的事件驅動方式
Netflix 主要透過監控表格狀態的變化來觸發優化任務：
*   **快照提交事件 (Snapshot Commit Events)**：每當 Iceberg 表格提交新的快照時，元數據存儲 (Meta Store) 會發出一個事件到 **SQS**。
*   **Autotune 監聽**：優化服務 Autotune 會監聽這些 SQS 事件，一旦得知有新的提交，便會根據配置判斷是否需要啟動 Spark 任務進行檔案合併 (Compaction) 或重寫。
*   **數據在地化觸發 (Autolift)**：Autolift 服務會掃描傳入的快照流，識別需要從遠端區域搬移到本地區域的檔案，並利用 Iceberg 的 API 進行原子性替換。

### 2. Apple 提到的核心事件類型
Apple 在其表格管理服務 (TMS) 中，強調了從「靜態排程」轉向「實時事件」的重要性，並定義了四種核心事件來觸發任務：
*   **寫入量 (Ingestion Volume)**：根據數據寫入的大小來判斷。
*   **寫入次數 (Ingestion Count)**：根據寫入發生的頻率。
*   **小檔案數量 (Small File Count)**：當累積的小檔案達到一定閾值時觸發。
*   **元數據變更 (Metadata Changes)**：監控元數據目錄 (Catalog) 的變化。

這些事件會直接觸發資料壓縮 (Compaction)、快照過期處理 (Snapshot Expiry) 以及孤兒檔案清理 (Orphan File Removal) 等任務。

### 3. 事件驅動的架構組成
兩者在架構設計上都有對應的組件來支援事件處理：
*   **控制層的事件管理器 (Event Manager)**：Apple 的架構中包含一個事件管理器，負責提供實時洞察，並針對各種條件發出詳細事件，以實現自動化響應。
*   **目錄整合 (Catalog Integration)**：Apple 指出可以從 Catalog 發送事件來觸發 Spark 任務，使系統能根據動態數據變化即時反應，而非僅依賴固定的時間表。

這種事件驅動的模式能讓維護工作更具**主動性 (Proactive)** 與**精確性**，確保表格始終處於最佳健康狀態，同時優化資源分配。

## 多說明不衝突提交：透過 REST Catalog，系統可以偵測到互不衝突的提交（例如：一個任務更新 Schema，另一個寫入數據檔案），從而避免不必要的任務失敗與重試。引擎層優化：Apple 提到，優化 Iceberg 核心引擎以增強併發寫入時的衝突解決能力，是未來處理高頻寫入與同時執行壓縮任務的關鍵方向。

針對您提到的 **「不衝突提交」** 與 **「引擎層優化」**，根據來源資料的詳細說明如下：

### 1. 不衝突提交 (Deconflicting Commits) 與 REST Catalog
在傳統的 Iceberg 實作中，提交通常基於狀態（State-based），如果兩個寫入者同時嘗試更新表格指標，其中一個往往會失敗並需要重試。**REST Catalog** 透過以下機制改善了這一點：

*   **基於變更的 API (Change-based API)**：REST Catalog 協議採用基於變更的設計，這使得系統能夠辨識不同類型的元數據更新，而非僅僅檢查整個表格狀態是否改變。
*   **偵測互不衝突的變更**：當多個任務同時發生時，如果它們修改的是元數據中不相關的部分，REST Catalog 可以偵測到這些變更並不會互相干擾。
    *   **具體範例**：一個任務正在進行 **Schema 更新**（修改欄位定義），而另一個任務正在 **寫入新的數據檔案**（增加數據）。
    *   **效益**：在舊有模式下，這兩者可能因競爭同一個提交鎖而導致其中一方失敗；但在 REST Catalog 下，系統知道這兩者不衝突，因此可以同時接受，避免了在高規模（例如數十個併發寫入者）環境下不必要的任務重試與資源浪費。

### 2. 引擎層優化 (Iceberg Core Engine Optimization)
Apple 強調，雖然建立外部的表格管理服務 (TMS) 很重要，但 **Iceberg 核心引擎** 本身的衝突解決能力也是關鍵的進化方向：

*   **應對高併發環境**：在數據寫入量極大、更新頻繁，且同時有背景維護任務（如資料壓縮 Compaction）運行的環境中，核心引擎面臨極大的衝突挑戰。
*   **優化衝突解決機制**：Apple 認為優化 Iceberg 內部的衝突解決邏輯是實現「真正自主運行」的核心。
    *   **具體目標**：提升引擎在處理「高頻寫入」與「併發維護任務」同時發生時的穩定性，確保系統在高度併發的數據湖環境下仍能保持高效運作。
*   **開源社群動態**：目前開源社群已經開始朝著增強核心引擎衝突處理能力的維度進行開發。

總結來說，**REST Catalog** 提供了一個更聰明的「通訊層」來過濾掉假性衝突，而 **引擎層優化** 則是從「資料結構處理層」提升 Iceberg 面對極限併發狀況時的韌性。

## 簡介一下 AutoComp 的 OODA

在 **AutoComp** 框架中，**OODA** 是一個用來對應資料緊實化（Compaction）工作流程的決策模型，其全稱為「**觀察、定位、決策、行動**」（Observe, Orient, Decide, Act）。此模型將複雜的緊實化過程拆解為四個模組化階段，讓系統能根據需求進行自定義與擴展。

以下是 OODA 模型各階段的詳細簡介：

*   **觀察 (Observe)**：
    此階段會從候選對象（如資料表或分區）中**提取相關統計數據 (Statistics)**。這些數據包括檔案層級的指標，或是特定於候選對象的表級或分區級統計。例如，統計候選對象中的檔案數量及其對應的大小。

*   **定位 (Orient)**：
    將觀察到的統計數據轉換為**特徵 (Traits)**，用來描述候選對象的當前狀態或未來潛力。特徵可分為兩類：描述緊實化**效益**的（如檔案數量減少預估、檔案熵），以及代表**成本**的（如計算成本）。這些特徵將作為後續排名與優先順序化的輔助依據。

*   **決策 (Decide)**：
    根據預定義的排名函數對候選對象進行排序，生成一個有序的緊實化清單。AutoComp 支持多種排名策略，例如在資源受限的環境下，使用**多目標優化 (MOOP)** 來平衡檔案減少量（效益）與計算預算（成本）之間的拉鋸。

*   **行動 (Act)**：
    最後一個階段是**調度並執行**選定的緊實化任務。根據叢集的配置，緊實化任務可以在原叢集執行，或是移交給專門的緊實化叢集，以減少對用戶正常工作負載的性能影響。

**OODA 模型的核心優勢：**
1.  **模組化與擴展性**：各階段互不干涉，允許用戶根據需求整合新的緊實化策略或決策標準。
2.  **靈活性**：支持在各階段之間加入**過濾機制**（例如跳過過小的表或近期頻繁寫入的表），並設有**回饋迴圈 (Feedback Loop)**，讓系統能根據執行結果不斷優化緊實化過程。
3.  **支持多種觸發模式**：無論是「寫入後優化」（Optimize-After-Write）的推送模式，還是「定期緊實化」（Periodic Compaction）的拉取模式，都能無縫整合進此工作流中。

## 多說明 無論是「寫入後優化」（Optimize-After-Write）的推送模式，還是「定期緊實化」（Periodic Compaction）的拉取模式

AutoComp 框架設計了兩種主要的執行觸發機制，以滿足現代資料湖對靈活性與資源管理的雙重需求（功能需求 FR3）。以下是這兩種模式的詳細說明：

### 1. 寫入後優化 (Optimize-After-Write)：推送 (Push) 模式
此模式的核心在於「即時性」，旨在數據剛寫入後便立即處理碎片化問題。

*   **觸發機制**：利用整合在運算引擎（如 Apache Spark）中的**鉤子（Hooks）**。當引擎完成資料修改或大規模寫入時，會主動將訊息「推送」給自動緊實化服務。
*   **決策邏輯**：如果觀察到的特徵值（如小檔案數量）超過了預設閾值，系統會立即觸發緊實化任務，或通知服務重新計算候選對象的排名。
*   **優點**：
    *   **效能即時提升**：能迅速遏制小檔案的增長，確保後續查詢能立即享有優化後的佈局。
    *   **適合高頻更新**：對於需要維持在最佳狀態的關鍵表特別有效。
*   **挑戰**：若選擇「立即觸發」，通常需要擁有幾乎不受限的計算預算，否則頻繁的緊實化可能會與正常的寫入負載競爭資源。

### 2. 定期緊實化 (Periodic Compaction)：拉取 (Pull) 模式
此模式將緊實化視為一種「例行維護」，強調可預測性與全域優化。

*   **觸發機制**：AutoComp 以**獨立服務**的形式運行（例如整合在 OpenHouse 控制平面中），不需要修改引擎驅動程式。服務會定期（如每小時或每天）主動從目錄或叢集狀態中「拉取」候選表的統計數據進行評估。
*   **執行策略**：LinkedIn 的 OpenHouse 實踐中，最初採用的是**固定間隔（每天一次）**的調度策略，選擇排名最高的前 $k$ 個候選者進行優化。
*   **優點**：
    *   **成本管理可控**：提供可預測的計算成本支出，不會因突發的寫入峰值導致預算失控。
    *   **錯峰執行**：可以安排在**離峰時段**（叢集利用率較低時）執行，避免干擾活躍的工作負載。
*   **缺點**：檔案碎片可能會在兩次執行間隔之間累積，導致某些時段的查詢效能暫時下降。

### 兩種模式的整合與靈活性
AutoComp 的設計允許作為一個獨立組件，同時支援這兩種操作：

*   **自適應性**：系統可以根據表的重要性或工作負載特性，對某些表使用「推送」模式確保即時性，對另一些表則使用「拉取」模式來節省資源。
*   **模組化處理**：無論觸發來源為何，所有候選對象都會進入相同的 **OODA 工作流**，確保決策過程的一致性與透明度（非功能需求 NFR1, NFR2）。
